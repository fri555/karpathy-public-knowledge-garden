# 20260526 · 综述:从二手解读到一手源 —— CLAUDE.md 神文件的两次 ingest 教训

> 本综述对比同一天 ingest 的两份资料(知乎二手文章 vs GitHub 一手仓库),记录"二手源失真"的具体形态,沉淀为知识库未来 ingest 的方法论教训。

## 触发场景

2026-05-26,先生先后给出两个目标:
1. 录入知乎专栏 [[20260526-知乎-CLAUDE.md神文件让ClaudeCode听话|《2.3K 小文件狂揽 10 万颗星》]]

两次 ingest 间隔约 15 分钟,完美对照"二手 vs 一手"的信息密度差异。

## 最严重的丢失:第 4 条原则

知乎文章基于"三宗罪"框架叙述,自然只能讲对应的前 3 条原则。但 [[CLAUDE.md]] 实际包含的第 4 条 [[Goal-Driven Execution]] 是**完全不同性质的原则**:

- 前 3 条是**减法**——告诉 LLM 别做什么(防错)
- 第 4 条是**加法**——告诉 LLM 优势在哪(放权)

二手解读因为框架预设(三宗罪),把第 4 条"放权激发优势"的关键洞察完全过滤掉了。这是**框架化叙述的代价**——用一个已有概念去套新事物,容易丢掉"不在概念里的部分"。

## 数据时效问题

知乎文章发表后,GitHub star 从 10.2 万涨到 15.5 万,涨幅约 **52%**。这提醒:
- 任何**数字类事实**(star、用户量、估值、版本号)二手源都自带时间衰减
- 一手 API(GitHub API、知乎 API)能拿到当下值,**ingest 时务必优先一手**
- 本知识库应规范:数字陈述后必须标 `(截至 YYYY-MM-DD)`

## 教训(沉淀为本知识库未来 ingest 的方法论)

### 教训 1:**优先一手源**
看到二手解读时,**先查是否有一手源可达**:
- 文章讲 GitHub 项目 → 直接抓仓库 API + README
- 文章讲推文 → 抓推文原文
- 文章讲论文 → 抓 arXiv
- 二手源只用于"了解大众解读视角",不用于"建立事实基础"

### 教训 2:**警惕框架化叙述**
二手作者用某个框架(如"三宗罪")组织叙述,会自动过滤掉"框架外"的信息。一手源往往**没有框架包装**,信息更原生。

### 教训 3:**数字事实必标时间**
一切随时间变化的数字(star/用户/估值/版本)陈述时必须带 `(截至 YYYY-MM-DD)`,避免后续读者误以为是当下值。

### 教训 4:**ingest 完成后回头检查"有没有 surprise"**
- 一手源 ingest 完,反向回看二手摘要,看哪些"应有未有"
- 这些"surprise"往往是最珍贵的洞察
- 本次的"第 4 条原则"就是回头检查发现的

### 教训 5:**新发现要修正既有页**,不要新建并行页
- 本次发现"三宗罪 → 4 个观察"修正后,选择保留原页名 + 加修正说明,而非新建并行页
- 避免知识库分裂、孤儿页、矛盾沉积
- 修正时在页面顶部用 `⚠️ 命名修正(YYYY-MM-DD ...)` 块标注
