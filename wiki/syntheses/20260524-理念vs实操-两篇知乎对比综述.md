## 关键差异点（⚠️ 待裁决）

### 1. Agent 后端选择

| | 第一篇推荐 | 第二篇推荐 |
|---|---|---|
| 后端 | [[Claude Code]] / [[Codex]] / OpenCode | [[Qwen Code]] |
| 动机 | 跟随 Karpathy 原始方案 | 国内网络可达性 |

**本知识库的选择**：方案 A（[[Hana]] 直接扮演 Agent），底层是 Claude 模型，但不走 Claude Code CLI。等于两者之外的第三条路。

### 2. Agent 入口

| | 第一篇 | 第二篇 |
|---|---|---|
| 入口 | CLI（命令行调用 Agent） | Obsidian 侧边栏（[[obsidian-agent-client]]） |
| 优势 | 灵活、无插件依赖 | 集成度高、所见即所得 |
| 劣势 | 切换上下文 | 多一层第三方插件依赖 |

### 3. AGENT.md 的获取方式

| | 第一篇 | 第二篇 |
|---|---|---|
| 路径 | 自己读 Gist 自己写 | 让 Agent 读 Karpathy 方法论文档自动生成 |

**本知识库的选择**：折中——Hana 读完两篇 + 行业理解后手写 AGENT.md，而非让 Agent 盲生成。

## 隐藏信息

- 第二篇提到 B 站 BV1p4DeB8ECi 有更详细教程，**包含 up 主自制的 skill 和 AGENT.md**。这是值得追查的高价值线索 → 见 [[#待深入]]

## 给本知识库的启发

1. **方案 A 是两篇之外的简化路径**，没有任何插件、CLI 依赖，但本质遵循同一方法论
2. **若未来要升级到方案 B，第二篇是直接的操作清单**——无需再调研
3. **B 站 up 主的 AGENT.md 值得对比**——可能对本知识库 AGENT.md 有改进启发

## 我的批注 / 思考

- 两篇文章构成有趣的"理念-实操"配对，这其实是值得专门做一类综述的模板：**任何重要方法论都应同时收录理念篇和落地篇**
- [[菠萝吹雪]] 篇有个隐含价值：示范了 **Agent 自建结构** 的可能性。理论上 AGENT.md 也可以让 Agent 自己写、自己改——但这与本知识库 §5 "迭代本协议要走流程" 的约束存在张力，值得未来讨论
