# Agent 工作台能力分层

> 一句话定义：把功能完备的 AI Agent 工作台拆成从用户界面到模型、工具、数据、权限、观测和运营的多层能力图谱，避免把“Agent”误解成一个 prompt 或一个聊天接口。

## 七层模型

### 1. 体验层

用户实际看到的界面：

- 工作台首页
- 会话页
- 任务时间线
- 文件/知识库视图
- 产物预览
- 设置与权限

这一层决定“好不好用”。

### 2. 任务层

把用户目标变成可追踪工作单元：

- conversation
- task
- step
- status
- retry
- approval

没有任务层，Agent 就只是聊天；有了任务层，用户才能知道“现在做到哪了”。

### 3. Agent 编排层

负责计划和执行：

- Planner：规划步骤。
- Executor：执行步骤。
- Router：决定调用哪个模型或工具。
- Memory Manager：选择上下文。
- Guardrails：限制危险行为。

### 4. 工具层

把 Agent 连接到真实世界：

- 文件读取
- 网页搜索
- 数据库查询
- 代码执行
- 第三方办公 API
- MCP Server

工具层的关键不是“多”，而是“可声明、可授权、可审计、可失败恢复”。

### 5. 知识与上下文层

让 Agent 能使用历史和资料：

- 消息历史
- 文件解析
- 摘要
- RAG
- 用户偏好
- 项目记忆
- Just-in-Time 检索

这一层与 [[上下文工程]] 直接相关。

### 6. 数据与基础设施层

让系统能长期运行：

- PostgreSQL：业务数据。
- 对象存储：原始文件。
- 向量数据库：语义检索。
- Redis / Queue：后台任务。
- 部署平台：运行服务。

### 7. 治理与运营层

让系统可控、可信、可持续：

- 权限
- 审计日志
- 成本统计
- 速率限制
- Evals
- 监控告警
- 数据保留策略

## 与相关概念的关系

- 与 [[AI Agent 工作台]]：本页是其架构拆解。
- 与 [[Agent工程化兜底]]：治理与运营层就是兜底的制度化实现。
- 与 [[验证瓶颈]]：任务层、日志和 Evals 是缓解验证瓶颈的工具。
- 与 [[../entities/MCP|MCP]]：工具层可用 MCP 标准化外部连接。
- 与 [[RAG]]：RAG 位于知识与上下文层，不应被误认为完整工作台。

## 典型误区

- 把模型 API 接上聊天框，就以为有了 Agent 工作台。
- 先做很多工具，但没有权限、日志和错误恢复。
- 先做复杂多 Agent，但用户连任务状态都看不见。
- 把 RAG 当作全部智能，而忽略执行和产物沉淀。
