# 反馈闭环写入文件

> 一句话定义：Agent 的反馈**必须具体到可以直接写进可持续加载的文件**，否则下一轮 Agent 会再犯同样错误——这是 Agent 系统"真正进化"的唯一通道。

## 核心要义

[[Shubham Saboo]] 在 30 天复盘中反复强调的执行铁律。很多人搭完 Agent 就让它自己运行，过几天发现效果没提升，以为系统没用——但问题往往不在模型，而在于反馈没有真正进入系统。

闭环流程：
1. Agent 输出 → 人工反馈
2. **反馈写入记忆文件或技能文件**
3. 下一轮任务加载更新后的文件
4. Agent 行为改变

断环的典型情况：只说"太长了，把第一段删掉"但不写入文件 → 下次同样错误。

## 关键要素

- **反馈必须具体**：不能"改好一点"，必须具体到足以写成规则（来源: [[20260528-36氪-Saboo30天6Agent复盘]]）
- **写入 ≠ 聊天**：聊天记录不会跨 session 持久化（来源: [[20260528-36氪-Saboo30天6Agent复盘]]）
- **两类文件**：记忆文件（偏好 BAD/GOOD）+ 技能文件（操作 SOP）——见 [[记忆文件vs技能文件]]

## 与相关概念的关系

- 与 [[纠错式Prompt工程]]：纠错式是方法论，反馈闭环写入是具体执行手段
- 与 [[Hooks确定性执行]]：Hooks 是"建议变法律"的强制闭环，反馈闭环写入是"经验变规则"的软闭环
- 与 [[原则化指令]]：反馈闭环的最终产出就是原则——本库 AGENT.md 也在每次 ingest 中进化

## 典型应用 / 案例

- Saboo 的 Kelly Agent：每条推文反馈都写入 BAD/GOOD 文件 → 第 20 天可直接发推 (来源: [[20260528-36氪-Saboo30天6Agent复盘]])
- 本知识库：每次 ingest 后更新 AGENT.md / index / log → 就是反馈闭环写入的 Wiki 版 (来源: 自我观察)

## 争议 / 局限

- 如果反馈频率过高（第一周的"擦屁股"阶段），写入成本可能超过手动完成成本
- 需要 Agent 严格遵守"加载更新后的文件"这一步——模型有时会忽略文件更新
