# 上下文膨胀与压缩

> 一句话定义：长期运行的 Agent 会因为历史记录不断积累导致上下文膨胀（如 161K token），响应变慢、输出降质——必须**每两周强制压缩**到合理量级（~40K），像软件的代码重构一样定期维护。

## 核心要义

[[Shubham Saboo]] 在 30 天 6 Agent 实战中亲身经历的工程问题。Kelly Agent 上下文一度 161K，Dwight 156K——直接的副作用：响应变慢、质量下降。压缩方法是**只保留当前真正有用的规则和记忆**，其余归档。

Saboo 给自己定的硬阈值：**>15 万 token 必须强力压缩**。

## 关键要素

- **膨胀速度**：6 个 Agent 在 30 天内多个超过 150K token（来源: [[20260528-36氪-Saboo30天6Agent复盘]]）
- **压缩比例**：Kelly 161K → 40K（4 倍压缩） / Dwight 156K → 43K（来源: [[20260528-36氪-Saboo30天6Agent复盘]]）
- **频率**：每两周一次（来源: [[20260528-36氪-Saboo30天6Agent复盘]]）
- **类比**：软件项目的代码重构——不清理就臃肿（来源: [[20260528-36氪-Saboo30天6Agent复盘]]）

## 与相关概念的关系

- 与 [[记忆文件vs技能文件]]：拆分是预防膨胀的手段之一
- 与 [[CLAUDE.md四层架构]]：Shareuhack 的 "CLAUDE.md > 300 行就拆 .claude/rules/" 是同一思路的不同表达
- 与 [[REM 睡眠式记忆整合]]：Anthropic Dreams 的"睡眠时整合记忆"在工程语义上就是上下文压缩的自动化版本

## 典型应用 / 案例

- Saboo Mac Mini fleet 的两周一压缩节奏 (来源: [[20260528-36氪-Saboo30天6Agent复盘]])
- 本知识库 Maintain 工作流 = 每隔一段时间审计、剪枝、整合 (来源: 自我观察)

## 争议 / 局限

- 压缩什么、保留什么需要人工判断，难以全自动
- 压缩太狠可能丢失 Agent 已经习得的能力
