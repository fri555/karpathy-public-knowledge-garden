# LLM 编程三宗罪

> ⚠️ **命名修正(2026-05-26 一手源补全后)**:
> 严格说应该叫 **"LLM 编程的 4 个 Karpathy 观察"** —— 前 3 个是"缺陷"(三宗罪),第 4 个是"优势"(循环执行能力)。
> 知乎二手文章只讲了前 3 个,因此叫"三宗罪",本页保留原名以保留历史叙述,但内容已按一手源补全为 4 个。
> 别名:[[LLM 编程 4 个观察]]

> 一句话定义:[[Andrej Karpathy]] 在 X 推文中提炼出的 **LLM 写代码 4 个核心观察**——3 个缺陷 + 1 个优势——共同构成 [[CLAUDE.md]] 4 条原则的设计基础。

## 核心要义

LLM 在写代码场景下,有 3 个跨模型代际、跨厂商的共同毛病,以及 1 个被低估的优势。前 3 条之所以是"祖传",根源在于 LLM 的"猜测惯性"和"过度抽象惯性";第 4 条揭示 LLM 真正擅长的方向是"循环执行至目标达成"。

[[forrestchang]] 据此设计了 [[CLAUDE.md]] 神文件,用 4 条原则**一对一**矫正/激发——前 3 防错,第 4 激发优势。

## Karpathy 4 个原始观察(英文推文中文翻译,来源 README.zh.md)

### 观察 1:缺陷 — 默默做错误假设
> "模型会代你做错误假设,然后不假思索地执行。它们不管理自身的困惑,不寻求澄清,不呈现矛盾,不展示权衡,在应该提出异议时也不反驳。" (来源: [[20260526-GitHub-andrej-karpathy-skills原仓库]])

→ 对应原则:[[Think Before Coding]]

### 观察 2:缺陷 — 把简单问题复杂化
> "它们真的很喜欢把代码和 API 搞复杂,堆砌抽象概念,不清理死代码……明明 100 行能搞定的事情,非要实现成 1000 行的臃肿架构。"

→ 对应原则:[[Simplicity First]]

### 观察 3:缺陷 — 乱改不相干的代码
> "它们有时仍会改动或删除自己理解不足的代码和注释,即使这些内容与任务本身无关。"

→ 对应原则:[[Surgical Changes]]

### 观察 4:**优势** — 循环执行至目标达成
> "LLM 非常擅长循环执行直到达成特定目标……不要告诉它该做什么,**给它成功标准**,然后看着它完成。"

→ 对应原则:[[Goal-Driven Execution]]

## 关键洞察(本次修正最大收获)

**不是"三宗罪 → 四原则凑数"**,而是 **"Karpathy 4 个观察 → 4 个原则" 完整对应**:
- 3 条防错原则 + 1 条激发优势原则
- 前 3 是"减法"(别犯错),第 4 是"加法"(用好特长)
- 知乎二手文章漏掉了第 4 条,正是漏掉了"放权给 LLM"这个最反直觉的洞察

## 与相关概念的关系

- **对应解药**:[[Think Before Coding]] + [[Simplicity First]] + [[Surgical Changes]] + [[Goal-Driven Execution]]
- **载体**:[[CLAUDE.md]] 神文件
- **哲学呼应**:与本知识库 [[../../AGENT|AGENT.md]] 的红线"不确定时停下来问人类"是同一思想——都反对 LLM 的"猜测惯性"
- **与 [[Ingest 录入]] 的关系**:Agent 在 ingest 时也可能犯三宗罪,本知识库的工作流就是在防它们

## 典型应用 / 案例

- [[CLAUDE.md]] 神文件:把 4 个观察对应的原则写进 markdown,放仓库根目录,[[Claude Code]] / [[Cursor]] 启动时自动加载 (来源: [[20260526-GitHub-andrej-karpathy-skills原仓库]])
- 用过 [[Claude Code]] 写过几段代码的开发者,普遍能识别出前 3 个毛病——这是 CLAUDE.md 能爆红的群众基础

## 争议 / 局限

- 这是 Karpathy 的个人观察,不是学术结论,但被实践广泛印证
- 4 个观察并未穷尽 LLM 编程的所有问题(例如忘记上下文、循环踩坑、过度自信回答未知 API 等)
- "把简单问题复杂化"在某些场景下"扩展性优先"存在张力——不是绝对原则
