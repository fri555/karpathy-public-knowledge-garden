# Agent 工作台产品化

> 一句话定义：Agent 工作台产品化，是把“模型能做”变成“用户敢用、可追踪、可恢复、可持续运营”的过程；核心不是再加一个 prompt，而是补齐信任、控制、反馈和运营系统。

## 核心要义

Demo 的成功标准是“看起来能跑”，产品的成功标准是“真实用户反复使用仍然可控”。

因此 Agent 工作台产品化必须回答：

- 用户如何知道 Agent 正在做什么？
- Agent 失败后如何恢复？
- 哪些动作需要用户确认？
- 如何限制工具权限？
- 如何记录成本和 token？
- 如何判断结果质量？
- 如何让用户下次继续同一个工作？

## 产品化清单

### 可解释

- 显示计划。
- 显示步骤。
- 显示工具调用。
- 显示引用来源。
- 显示失败原因。

### 可控制

- 敏感动作二次确认。
- 支持暂停/取消任务。
- 支持重试单个步骤。
- 支持选择模型和工具范围。

### 可恢复

- 长任务进入后台队列。
- 中间状态持久化。
- 工具失败可重试。
- 会话刷新后不丢进度。

### 可审计

- 保存 tool_calls。
- 保存输入输出。
- 保存执行时间、成本和错误。
- 保存谁授权了什么。

### 可运营

- 成本统计。
- 速率限制。
- 用户反馈。
- 质量评测。
- 模板和工作流复用。

## 与相关概念的关系

- 与 [[Agent工程化兜底]]：产品化清单中的权限、审计、确认、重试都是工程兜底。
- 与 [[验证瓶颈]]：可解释和可审计降低人类验证成本。
- 与 [[Goal-Driven Execution]]：产品化需要清晰成功标准和持续验证。
- 与 [[上下文工程]]：可恢复和长期会话要求管理 context 的生命周期。

## 好用的标准

一个好用的 Agent 工作台，不是“Agent 自主性最高”，而是：

- 用户知道它在做什么。
- 用户知道什么时候该介入。
- 系统出错时不吓人。
- 结果能沉淀，不会问完即忘。
- 工具调用有边界。
- 复杂任务能分阶段完成。
