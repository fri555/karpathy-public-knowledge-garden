# 验证瓶颈

> 一句话定义：AI 让代码/内容生成成本急剧下降后，真正稀缺的环节从“生成”迁移到“验证、审查、判断和负责”。

## 核心要义

[[davidycwei]] 的关键判断是：**生成能力的爆炸式增长与验证能力的相对停滞之间的张力，是 AI 时代软件工程的核心矛盾** (来源: [[../summaries/20260306-知乎-从第一性原理思考AgenticEngineering]])。

过去瓶颈是“写不出来 / 写得慢”；现在瓶颈是：

- AI 一小时能生成几百上千行代码，但人类仍要判断它是否符合设计意图。
- 代码表面能跑，不代表边界条件、安全性、一致性、可维护性正确。
- 审查信息量暴增，工程师的认知带宽可能被 AI 输出淹没。

## 关键要素

- **生成成本下降**：AI 负责样板代码、测试、Mock、局部实现。
- **验证成本未同步下降**：正确性仍需工程判断、测试、边界分析和代码审查。
- **责任不转移**：Agent 生成代码，但系统责任仍在人类/团队。
- **需要工程化解法**：Spec、测试、门禁、分阶段审查，而不是“多问几遍 AI”。

## 与相关概念的关系

- 与 [[Agentic Engineering]]：验证瓶颈是 Agentic Engineering 要解决的核心矛盾之一。
- 与 [[Vibe Coding]]：Vibe Coding 主动放弃验证，因此只适合低风险原型/个人工具。
- 与 [[Goal-Driven Execution]]：用可验证目标和测试循环部分缓解验证瓶颈。
- 与 [[Agent工程化兜底]]：用确定性门禁把部分验证从人脑迁移到脚本/CI。
- 与 [[Spec-First工作流]]：Spec 是验证的前置锚点，没有 spec 就无法判断是否偏离目标。

## 典型应用 / 案例

- TDSQL HBase conditional mutation 案例中，Agent 在编码前先生成测试计划，覆盖正确性、异常路径、并发一致性 (来源: [[../summaries/20260306-知乎-从第一性原理思考AgenticEngineering]])。
- 复杂任务先澄清成功标准与非目标，避免“能跑但不对”。

## 争议 / 局限

- 有些任务的验证标准难以形式化，如 UI 审美、文案质量、产品策略。
- 过度验证会拖慢轻量任务，因此需要风险分级，而不是所有任务都上重流程。
