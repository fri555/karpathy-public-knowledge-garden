# Goal-Driven Execution

> 一句话定义:[[CLAUDE.md]] 神文件第 4 条原则——**定义成功标准,循环验证直到达成**(Define success criteria. Loop until verified.)。是 4 条原则中**唯一一条"激发 LLM 优势"而非"防止 LLM 犯错"**的原则。已被工程界 [[Codex]] `/goal` 模式产品化。

## 核心要义

[[Andrej Karpathy]] 原文:**"LLM 非常擅长循环执行直到达成特定目标……不要告诉它该做什么,给它成功标准,然后看着它完成。"**

具体操作:把指令式任务转化为可验证目标 (来源: [[20260526-GitHub-andrej-karpathy-skills原仓库]]):

| 不要这样 | 转化为 |
|---|---|
| "添加验证" | "为无效输入编写测试,然后让它们通过" |
| "修复 bug" | "编写重现 bug 的测试,然后让它通过" |
| "重构 X" | "确保重构前后测试都能通过" |

多步任务的模板:
```
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
3. [步骤] → 验证: [检查]
```

**精髓**:Strong success criteria let you loop independently. Weak criteria ("make it work") require constant clarification.

## 关键要素

- **可验证性**:成功标准必须能用代码/测试/输出来检查,不能是"看起来对"
- **循环授权**:有了强标准,LLM 才敢自己跑循环——这才是 LLM 的优势区
- **测试优先思维**:"先写测试再实现"反过来给 LLM 用,效果惊人
- **断点续跑**:好的循环模式应该支持断掉再开,Agent 接续探索(实际产品案例:Codex `/goal`)

## 工程实证:Codex `/goal` 模式

[[卡尔的AI沃茨]] 在 [[淘金小镇]] 项目中提供了**最强的工程验证案例** (来源: [[20260527-知乎-卡尔-淘金小镇Skill榜单情报项目]]):

| 路径 | 结果 |
|---|---|
| 浏览器自动化 + 复杂提示语 | **一周三天失败** |
| 升级到 GPT 5.5 high(顶级模型) | 失败率仍超高 |
| **Codex `/goal` 循环 30 次,目标=找出依赖最少的方案** | **第 5 小时发现 Convex API,问题解决** |

**关键观察**:
- **算力 ≠ 成功率**:高级模型救不了弱目标
- **强目标 + 普通模型 + 循环 > 弱目标 + 顶级模型 + 复杂 prompt**
- **prompt 越详细 ≠ 越成功**,有时反而限制 Agent 探索

## 与相关概念的关系

- **不同于** [[LLM 编程三宗罪]] 的"防错"思路——这是"放权"思路,**激发 LLM 的循环能力**
- **直接落地形态**:[[Goal 模式自动化爬虫]] —— 在爬虫工程的具体形态
- **配合**:[[复刻用户视角拿数据]] —— Agent 探索过程中应优先找 API 而非 DOM
- **同源思想**:类似 TDD(测试驱动开发),但目标对象是 AI
- **延伸到本知识库**:ingest/maintain 流程可以从"步骤式"改成"目标 + 验证"模式:
  - 旧:`步骤 1 → 步骤 2 → ...`
  - 新:`目标 = 影响 5-15 页 + 0 矛盾 + log 已写 → 循环到达成`
- **属于**:[[原则化指令]] 范式

## 争议 / 局限

- 不是所有任务都有清晰可验证的标准(如 UI 设计、文案创作)
- 编写好的成功标准本身就是高难度技能,普通用户写不出强标准
- 循环验证有 token 成本,简单任务用此原则可能浪费资源
- 实践经验:循环上限要给(如 30 次),否则成本失控
