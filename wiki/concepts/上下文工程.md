# 上下文工程

> 一句话定义：[[../entities/Anthropic|Anthropic]] 官方提出的 Agent 时代核心范式——从 [[原则化指令|prompt engineering]] 进化到对 Agent 运行时全生命周期 context 的 curation 与维护，目标是"找到最小高信号 token 集合，最大化期望输出"。

## 核心要义

Context engineering 不只是写 prompt，而是管理 LLM 在每一轮推理时看到的**所有 token**——system instructions、tools、MCP、外部数据、消息历史、note、子 Agent 输出等。

核心原理：LLM 有 [[注意力预算|有限的注意力预算]]，context 越长 [[上下文腐烂|越腐烂]]——所以 context 必须被视为有限且边际收益递减的资源。

与 prompt engineering 的区别：
- Prompt engineering：一次性写好 system prompt（离散任务）
- Context engineering：每一轮 inference 都在动态决定下一个 token 集合（迭代任务）

## 关键要素

- **System prompt 的"正确海拔"**：既不脆性硬编码 if-else，也不空泛无信号；用 XML/Markdown 分段组织 (来源: [[../summaries/20260929-Anthropic-Effective context engineering for AI agents]])
- **工具高效原则**：返回 token-efficient 信息，避免工具集膨胀 (同上)
- **Examples 是 pictures**：少而经典 > 多而琐碎 (同上)
- **Just-in-Time 检索**：见 [[Just-in-Time检索]] (同上)
- **长时域三技术**：Compaction（见 [[上下文膨胀与压缩]]）/ Structured note-taking / Sub-agent architectures (同上)

- **进化关系**：[[原则化指令|Prompt Engineering]] → 本概念（自然进化）
- **下位概念**：[[注意力预算]] / [[上下文腐烂]] / [[Just-in-Time检索]] / [[渐进式披露]] / [[上下文膨胀与压缩]]

## 争议 / 局限

- Just-in-Time 比预计算慢——需要工程权衡
- 何为"高信号"难以量化，多数靠经验调优
- 子 Agent 隔离虽好，但增加协调成本
