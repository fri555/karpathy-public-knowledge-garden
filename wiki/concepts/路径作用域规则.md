# 路径作用域规则

> 一句话定义：Claude Code 的 `.claude/rules/{name}.md` 文件**只在 Claude 处理对应路径的文件时才载入**——让 CLAUDE.md 从静态文件进化成"动态规则引擎"，大幅减少不相关规则浪费 context。

## 核心要义

[[Shareuhack]] 介绍的进阶功能。匹配逻辑基于子目录名：`.claude/rules/frontend.md` 对应 `src/frontend/` 相关文件。

解决的问题：CLAUDE.md 越写越长是普遍问题，但很多规则其实只在某些路径下相关（前端规则、后端规则、文档规则）。把它们拆到 `.claude/rules/` 既减少了 CLAUDE.md 体积，又保留了规则的精确触发。

## 关键要素

- **触发条件**：路径匹配 + 文件名匹配（来源: [[20260528-Shareuhack-CLAUDE四层架构实战]]）
- **常用场景**：前后端分流、不同子项目（monorepo）的不同规范、文档区与代码区的不同语调（来源: [[20260528-Shareuhack-CLAUDE四层架构实战]]）
- **拆分阈值**：CLAUDE.md > 300 行就该考虑拆 .claude/rules/（来源: [[20260528-Shareuhack-CLAUDE四层架构实战]]）

## 与相关概念的关系

- 与 [[CLAUDE.md四层架构]]：路径作用域是 CLAUDE.md 层的内部分级机制
- 与 [[Skills 2.0模块化]]：Skills 是"按工作流"模块化，路径规则是"按文件位置"模块化
- 与 [[原则化指令]]：让原则可以"局部生效"而不是全局污染

## 典型应用 / 案例

- 团队项目中前端/后端工程师不互相干扰 (来源: [[20260528-Shareuhack-CLAUDE四层架构实战]])

## 争议 / 局限

- 匹配逻辑基于子目录名，对复杂目录结构可能不够灵活
- 文档对路径匹配规则的描述不完全清晰
