# 纠错式 Prompt 工程

> 一句话定义：与其一开始设计完美 prompt，不如先写粗糙版让 Agent 跑起来，通过持续把具体反馈写入文件来迭代——**第 1 版很烂，第 10 版开始能用，第 30 版才稳定**，像管理新员工一样管理 Agent。

## 核心要义

[[Shubham Saboo]] 在 30 天 6 Agent 复盘中提出的核心方法论。传统 Prompt Engineering 追求"一次写对"，而纠错式承认：**Agent 的第一版输出必然不合格**，但只要把每一次纠正具体化并写入可持续加载的文件，系统就会逐轮进化。

关键洞察：模型本身不变（第 1 天 = 第 30 天），**变的是围绕模型构建的规则系统**——这和 Karpathy "给它成功标准，看着它完成"（[[Goal-Driven Execution]]）的精神完全一致。

## 关键要素

- **SOUL.md / CLAUDE.md**：Agent 行为设定的粗糙初版（来源: [[20260528-36氪-Saboo30天6Agent复盘]]）
- **BAD 文件**：记录被否定的模式（如"不用 bullet points"）（来源: [[20260528-36氪-Saboo30天6Agent复盘]]）
- **GOOD 文件**：记录表现最好的样本供 Agent 模仿（来源: [[20260528-36氪-Saboo30天6Agent复盘]]）
- **反馈写入文件，不是聊天**：反馈只存在对话记录中，下次 Agent 会再犯同样错误（来源: [[20260528-36氪-Saboo30天6Agent复盘]]）

## 与相关概念的关系

- 与 [[Goal-Driven Execution]] 的区别：GDE 是"定义成功标准循环验证"，纠错式是"让它先跑，错了就改文件"——前者偏预设，后者偏进化
- 与 [[原则化指令]] 的联系：纠错式的最终产出就是原则——BAD/GOOD 积累到稳定后自然凝练为原则
- 与 [[反馈闭环写入文件]] 的联系：纠错式是宏观方法论，反馈闭环写入是微观执行机制

## 典型应用 / 案例

- [[Mac Mini Agent Fleet]] 的 Kelly（运营 Agent）：第 1 天冗长套路化 → 第 10 天 emoji 消失 → 第 20 天可直接发推 (来源: [[20260528-36氪-Saboo30天6Agent复盘]])
- [[淘金小镇]] 的淘金 Skill 榜单扫描也是纠错式——用 /goal 模式让 Agent 自己迭代 (来源: [[20260527-知乎-卡尔-淘金小镇Skill榜单情报项目]])

## 争议 / 局限

- 依赖"反馈必须具体"——模糊反馈（"改好一点"）无法规则化，不具备复利价值
- 初期是负收益——修改 Agent 输出的时间比自己干还长，大多数人会在混乱期放弃
