# AI Agent 工作台

> 一句话定义：AI Agent 工作台是把自然语言、任务规划、工具调用、文件/知识库、产物沉淀和长期上下文整合在一起的生产力系统；它不是聊天机器人，而是面向真实工作的 Agent 操作台。

## 核心要义

AI Agent 工作台的价值不在“能聊天”，而在“能把工作推进到可验收的产物”。

最小闭环：

```text
用户目标 -> Agent 规划 -> 工具执行 -> 观察结果 -> 继续执行 -> 产物沉淀 -> 下次可续
```

这意味着它至少要同时处理三类状态：

- **对话状态**：用户说过什么，AI 回答过什么。
- **任务状态**：计划、步骤、进度、失败、重试、完成。
- **工作产物状态**：文档、文件、表格、代码、链接、外部系统记录。

## 关键能力

1. **自然语言入口**：用户用目标描述任务，而不是先找按钮。
2. **任务拆解**：把大目标拆成可执行步骤。
3. **工具系统**：通过工具连接文件、网页、数据库、办公软件和代码运行环境。
4. **上下文管理**：决定哪些历史、文件、工具结果进入当前推理。
5. **产物管理**：把结果保存成文档、任务、表格、页面或外部系统动作。
6. **可观测性**：记录 Agent 做了什么、为什么失败、花了多少成本。
7. **安全与权限**：控制工具可用范围、敏感动作审批、数据访问边界。

## 与相关概念的关系

- 与 [[Agentic Engineering]]：工作台是 Agentic Engineering 从个人编码实践走向产品平台的载体。
- 与 [[上下文工程]]：工作台的核心难题之一，是给 Agent 组织合适的运行时 context。
- 与 [[RAG]]：RAG 只解决“从知识中找相关材料”，工作台还要解决“执行任务和沉淀产物”。
- 与 [[../entities/MCP|MCP]]：MCP 是工作台连接外部工具和数据源的重要协议候选。
- 与 [[Agent工程化兜底]]：工作台要好用，必须把 LLM 的概率性包进确定性的工程边界。

## 典型模块

- Dashboard：工作入口、最近任务、快捷动作。
- Chat Workspace：会话与任务协作界面。
- Files：上传、解析、摘要、引用。
- Tasks：执行状态、步骤、日志、失败原因。
- Artifacts：产物预览、版本、导出。
- Integrations：第三方系统授权和工具连接。
- Settings：模型、权限、成本、团队配置。

## 争议 / 局限

- 过早追求“万能工作台”会拖垮第一版交付。
- 只做聊天会低估工具、权限、日志、任务队列的复杂度。
- Agent 自主性越强，用户信任和可解释性越重要。
