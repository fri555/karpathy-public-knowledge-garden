# 复刻用户视角拿数据

> 一句话定义:抓取网页数据时,**不要去抓 HTML 文本,而是找网页自己用的后端 API**(Convex / GraphQL / REST),"网页怎么问,你就怎么问"——这样拿到的数据**最接近用户实际看到的**,且远比 DOM 抓取稳定。

## 核心要义

网页只是数据的展示层,真正的数据在后端 API。直接抓 HTML 有 N 个坑:
- 页面结构改了你就废了
- 异步加载/无限滚动抓不全
- 反爬识别浏览器特征
- 渲染后内容跟原始数据有信息损失

**找到 API 后,你就站在了"和用户看一样数据"的同一起点**,稳定性高 10 倍。

## 关键要素 (来源: [[20260527-知乎-卡尔-淘金小镇Skill榜单情报项目]])

1. **打开 Chrome DevTools / Network 面板**,看页面加载时调了哪些 XHR/Fetch
2. **找规律**:常见后端是 [[Convex]] / GraphQL / REST,query path 通常有业务名(如 `skills:listPublicPageV4`)
3. **分析参数**:`sort` / `dir` / `limit` / `cursor` / 业务字段
4. **处理分页**:常见模式 `nextCursor` 翻页 / `offset+limit` / `page` 参数
5. **认证策略**:有些 API 需要登录态(cookie),要看是否在请求头里

## 与相关概念的关系

- **优于**:DOM/HTML 抓取(传统爬虫思路)
- **优于**:浏览器自动化(虽然万能但脆弱)——见 [[淘金小镇]] 的失败教训(浏览器自动化"一周三天失败")
- **配合**:[[Goal-Driven Execution]] —— 让 Agent 自主探索 API 路径而非硬写 prompt
- **失败兜底**:Convex/GraphQL 找不到时再退回浏览器自动化

## 典型应用 / 案例

- [[淘金小镇]] 发现 ClawHub 用 Convex 后端,直接调 `skills:listPublicPageV4` 接口 (来源: [[20260527-知乎-卡尔-淘金小镇Skill榜单情报项目]])
- **可应用到本知识库的场景**:
  - 抓知乎专栏 → 走 `api.zhihu.com/v4/articles/{id}`(已验证可用,虽偶尔截断)
  - 抓 GitHub 仓库 → 走 `api.github.com/repos/...`
  - 抓 B 站视频信息 → 走 B 站开放 API
  - 抓飞书文档 → 走飞书 OpenAPI

## 争议 / 局限

- **API 没公开文档**:需要逆向,但通常有现成的工具(MITM proxy / Charles)
- **认证墙**:很多 API 要登录态,不一定能复刻
- **法律/ToS 风险**:复刻 API 可能违反平台条款,商用要谨慎
- **API 变更**:平台升级 API 你就废了——比 HTML 还快变
- **反向**:有些平台故意把 API 写得难懂(混淆参数名/cursor 加密)
