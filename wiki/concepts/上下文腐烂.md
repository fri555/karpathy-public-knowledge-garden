# 上下文腐烂

> 一句话定义：随着 context window 中 token 数量增加，LLM 准确召回信息的能力会下降——context 越长，"中间被遗忘"和"细节失真"越严重。

## 核心要义

来自 Chroma Research 的 needle-in-a-haystack 风格 benchmark 研究，[[../entities/Anthropic|Anthropic]] 将其作为 [[上下文工程]] 的物理基础引用。

所有模型都有这个特性——有的衰减温和，有的剧烈，但**没有模型能逃脱**。原因在 [[注意力预算]] 的架构约束。

实践含义：不要假设"塞进 context 的信息 = 模型能用的信息"——长 context 里的中段内容可能被模型实质性忽略。

## 关键要素

- **测量方式**：needle-in-a-haystack 在长 context 中藏一个事实，测模型能否准确召回 (来源: [[../summaries/20260929-Anthropic-Effective context engineering for AI agents]])
- **普遍性**：所有模型都有，差异在衰减曲线斜率 (同上)
- **不是悬崖**：是渐进衰减，不是某个 token 数突然失效 (同上)
- **位置编码插值**的副作用：让模型能处理超长序列，但 token 位置理解会退化 (同上)

## 与相关概念的关系

- 与 [[注意力预算]]：腐烂是预算耗尽的症状
- 与 [[上下文膨胀与压缩]]：压缩是对抗腐烂的手段
- 与 [[上下文工程]]：腐烂是本范式存在的理由
- 与 [[Just-in-Time检索]]：按需载入避免无关 token 占用 context

## 典型应用 / 案例

- 长会话 Agent 在第 50 轮可能"忘记"第 5 轮提到的关键约束
- [[../entities/Shubham Saboo]] 的 Mac Mini fleet 上下文 >150K 时输出降质（来源: [[../summaries/20260528-36氪-Saboo30天6Agent复盘]]）
- [[../entities/OpenClacky|OpenClacky]] 主动把历史压到 1 万 token 以内 (来源: [[../summaries/20260514-RubyChina-OpenClacky-Harness工程7个关键决策]])

## 争议 / 局限

- 不同任务受影响程度不同（检索 vs 推理）
- 模型迭代可能缓解但难以消除
