# Agent 工程化兜底

> 一句话定义：Agent 再智能，也是软件系统，**软件系统就需要边界和兜底**——用确定性的工程手段（Schema 校验、白名单、循环检测、Hooks）弥补 LLM 的概率性本质。

## 核心要义

[[20260528-掘金-MCP协议落地三条铁律|沙河 rain]] 在文章结尾的工程哲学声明，与 [[Shareuhack]] "Hooks 把建议变法律"完全同源——两个独立团队基于实战收敛到同一立场：

**LLM 的概率性 ≠ 系统的随意性**——前者是模型本质，不可改变；后者必须靠工程兜底消除。

这是对"AI 是万能的"派系的明确切割：不要试图用更好的 prompt 解决工程问题，工程问题需要工程解。

## 关键要素

- **确定性手段**：JSON Schema、白名单、循环检测、git hook、shell 执行（来源: [[20260528-掘金-MCP协议落地三条铁律]]）
- **位置原则**：把判断权从 LLM 拿回来交给确定性系统（来源: [[20260528-Shareuhack-CLAUDE四层架构实战]]）
- **态度声明**：Agent 是软件系统，遵守软件工程原则（来源: [[20260528-掘金-MCP协议落地三条铁律]]）

## 与相关概念的关系

- 与 [[Hooks确定性执行]]：Hooks 是兜底的具体机制
- 与 [[工具调用幻觉]] / [[字段级权限]]：都是兜底的具体应用
- 与 [[Goal-Driven Execution]]：GDE 是"循环验证"，兜底是"边界保护"——一个推进一个限制
- 与 [[LLM 编程三宗罪]]：三宗罪是要被兜底的对象

## 典型应用 / 案例

- 沙河 rain 的三层防幻觉 (来源: [[20260528-掘金-MCP协议落地三条铁律]])
- Shareuhack 的 PreToolUse / Stop Hook (来源: [[20260528-Shareuhack-CLAUDE四层架构实战]])

## 争议 / 局限

- 兜底过度会让 Agent 失去灵活性
- 兜底的工程成本可能让小项目得不偿失
