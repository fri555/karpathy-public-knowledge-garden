# 20260528 · 36氪转载智东西 · Shubham Saboo 30 天 6 Agent 复盘

> 一句话摘要：一个前 Google PM 用 30 天连续跑 6 个 Agent 给自己当员工，第 4 天差点团灭、第 4 周开始反过来提升效率——他把这 30 天提炼成"纠错式 Prompt 工程 + 记忆/技能文件二分 + 反馈闭环"三件事，模型不变是真的，变的是围绕模型的"系统"。

## 核心论点

1. **模型不会越用越聪明，"系统"会**。第 1 天和第 30 天的模型一模一样，真正进化的是规则文件、记忆文件和反馈闭环组成的工作流——**这是对"Agent = 模型 + harness"工程哲学的实操印证**
2. **纠错式 Prompt 工程（Corrective Prompt Engineering）**：与其设计完美 prompt，不如先写粗糙版 SOUL.md，让 Agent 跑、出错、把每条具体反馈写入文件——第 1 版很烂，第 30 版才稳定。**像管理新员工一样管理 Agent**
3. **反馈必须写入文件，不能停留在聊天里**。如果反馈只存在对话记录中，下一次 Agent 会再次犯同样错误。这是大多数人养 Agent 失败的最关键单点——"擦屁股"擦了一周以为没用，其实是没把擦的动作沉淀
4. **记忆文件 vs 技能文件必须拆开**。记忆文件 = 用户偏好（BAD/GOOD 样本），技能文件 = 操作 SOP。混在一起会让规则互相打架（第一周表达偏好 vs 第二周明确规则冲突）。**技能文件比记忆文件积累更快、效果更直接**
5. **上下文会"发胖"**。Kelly 一度膨胀到 161K token，必须**每两周强压缩**到 40K 量级，否则响应变慢、输出降质——这是 Agent 团队的"代码重构"
6. **Agent 团队的三阶段曲线**：混乱期（前 7 天，负收益）→ 稳定期（8-21 天，少量编辑）→ 复利期（第 4 周后，自动跑通）。**绝大多数人死在混乱期**
7. **不要一次上 6 个 Agent**。先把一个 Agent 做到稳定可用，再加第二个。两个 Agent 的协作要用最简单的"文件协作"（A 写文件，B 读文件）
8. **系统级 Agent 也要建**：他后来加了首席运营 Agent Monica 专门监控任务心跳（超过 26 小时未运行就触发重启）——**Agent 系统也需要 SRE**

## 原文金句

> "模型在第 1 天和第 30 天其实没有变化，不会越用越'聪明'。真正发生变化的，是围绕模型构建的系统——规则文件、记忆记录以及持续反馈形成的工作流程。"

> "反馈必须写入文件。如果反馈只存在对话记录中，下一次任务 Agent 就会再次犯同样的错误。只有当这些经验被写入可持续加载的文件，系统才会真正进化。"

> "把这个改好一点'这种话几乎不会留下任何有效积累。真正有用的反馈，必须具体到足以直接写进文件。"

> "第一版通常很普通，第十版开始能用，第三十版才会真正稳定。"
