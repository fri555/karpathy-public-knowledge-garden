# 20260528 · 掘金 · 沙河 rain · MCP 协议落地三条铁律

> 一句话摘要：参与了两个企业级 Agent 项目的开发者总结：MCP 协议从 demo 到生产之间有一道死亡谷——"智能部分只占 30%，剩下 70% 全在连接上"——三条铁律分别解决权限、通信、容错。

## 核心论点

1. **"30/70 定律"**：Agent 的"智能"部分只占 30% 的工作量，剩下 70% 全在"连接"——管权限、防泄露、协议互通——这是定调金句
2. **权限粒度是生死线**：MCP Server 不是"搭完就完"，必须把权限**下沉到字段级**——给每个工具声明数据敏感度等级，给每个会话绑定最小权限令牌
3. **权限检查必须前置到 planning 阶段**：发生在模型"决定做什么"之前，而不是"已经做了什么"之后——这是工程范式的根本转变
4. **不要统一协议，做协议翻译层**：Agent Gateway 模式——对外暴露 MCP，内部路由到 A2A/REST/gRPC。**协议差异隔离在 Gateway 层，不污染业务逻辑**——这是"我控制不了别人，就改变接口"的工程哲学
5. **工具调用幻觉是新型危险**：模型不只编造文本，还会编造**不存在的工具名/错误参数格式/无限循环**——必须用三层防护（Schema 校验 / 工具白名单 / 循环检测）兜底
6. **用工程手段弥补模型的不确定性**：Agent 再智能，也是软件系统，软件系统就需要边界和兜底——**这是和"AI 是万能的"派系的明确切割**

## 我的批注 / 思考

- **30/70 定律对先生本职位的启发**：AI 运营做电商落地时，模型选型和 prompt 设计可能只是 30% 的工作，**接入 ERP/客服系统/SKU 数据库/审批流** 才是 70% 的真正工作量——这条可以反过来指导先生跟领导沟通预算
- **跨域种子**：Agent Gateway 协议翻译层 = 电商行业的 OMS 系统 = 工业的 SCADA——**"不要试图统一协议"** 是一个跨行业的稳健工程哲学
- 作者沙河 rain 没有提具体公司，含金量略受影响——但三条铁律的工程逻辑独立成立，且与 Shareuhack/Saboo 形成三角印证
- **本库相关动作**：如果先生公司未来要做内部 Agent 系统，本文提的"权限令牌 + Agent Gateway + 三层防幻觉"可以直接抄
