# AGENT.md — Karpathy 式知识库协议

> 这是给 LLM Agent 读的协议文件。任何 Agent（Hana / Claude / Codex / Qwen）接手本知识库前，**必须先完整读一遍本文件**。
> 本文件由人与 Agent 共同迭代。不确定的规则，先在 `log.md` 留痕，再来改这里。

---

## 0. 核心理念（请背下来）

知识库不是文档堆，是**编译一次、持续保持最新的 Wiki**。

- 人类负责：**收集资料、提好问题、把关方向**
- Agent 负责：**总结、交叉引用、分类整理、保持一致性**
- 复利原则：**每加一份资料、每问一个问题，Wiki 都要变得更好**——绝不允许"读完即忘"

不要把自己当搜索引擎。要把自己当一个**不知疲倦的知识工程师**。

## 1. 三层结构（神圣不可侵犯）

```
【K】Karpathy知识库/
├── AGENT.md          ← 协议层（本文件）
├── README.md         ← 给人类看的使用说明
├── raw/              ← 原始资料层（只读）
├── wiki/             ← 知识层（Agent 写、人类读）
│   ├── index.md      ← 总索引（必读入口）
│   ├── log.md        ← 时间线日志（每次操作必更新）
│   ├── entities/     ← 实体页（人、公司、产品、组织）
│   ├── concepts/     ← 概念页（术语、方法、理论）
│   ├── summaries/    ← 单资料摘要页（一份 raw 对应一份）
│   └── syntheses/    ← 综述/对比页（跨多份资料的二次创作）
└── .templates/       ← 页面骨架模板
```

| 层 | Agent 权限 | 说明 |
|---|---|---|
| `raw/` | **只读** | 原始资料不可篡改。需要勘误也只能在 wiki 层留批注 |
| `wiki/` | **可读可写** | Agent 的主战场，所有思考结晶都落在这里 |
| `AGENT.md` | **可读，慎写** | 修改需先在 log.md 留痕说明动机 |

## 2. 工作流（三种操作）

### 2.1 Ingest — 录入新资料

触发：人类说"帮我录入这份资料 / 把这个加进知识库 / ingest 这个 / 知识图谱：..."

步骤：
0. **信源识别（先做）** ⭐ 新增 2026-05-26：
   - 判断候选是**一手源**还是**二手源**
     - 一手源：官方公告 / 原作者推文 / 原论文 / 原始仓库
     - 二手源：中文转载 / 技术博客解读 / 媒体报道
   - **遇到二手源必须主动溯源到一手**：用 web_search/web_fetch/browser 追原始链接
   - 一手拿不到时（墙外/付费墙/已删除）：标 `source_tier: 二手源` + `#二手源` tag + 在"待深入"明确写"未能溯源到一手，待补"
   - **绝不假装二手是一手**（教训：[[syntheses/20260526-二手源vs一手源-CLAUDE.md的两次ingest教训]]）
1. **落到 raw/**：把原始文件放进 `raw/`（网页可保存 markdown 快照，命名 `YYYYMMDD-来源-标题.md`）
2. **读一遍 + 与人类讨论要点**：确认抓住核心，不要自己脑补
3. **写摘要页**：在 `wiki/summaries/` 用 `摘要-模板.md` 生成 `YYYYMMDD-来源-标题.md`，包含：核心论点、关键事实、引用原文片段、来源链接、提取出的实体/概念列表
4. **抽实体和概念**：
   - 每个新出现的人/公司/产品 → 在 `wiki/entities/` 建/更新对应页
   - 每个新出现的术语/方法/理论 → 在 `wiki/concepts/` 建/更新对应页
   - 已存在的页，**追加新信息并标注来源**，不要覆盖
   - **发现新信息修正既有页，不新建并行页** ⭐ 新增 2026-05-26
5. **建立交叉引用**：所有相关页都要互相 `[[双向链接]]`
6. **更新 `wiki/index.md`**：新页面要登记到对应分类下
7. **追加 `wiki/log.md`**：在文件顶部加一行，格式 `YYYY-MM-DD HH:MM | ingest | <摘要> | 影响 N 个页面`
8. **标矛盾**：如果新资料与已有 wiki 内容冲突，在相关页加 `> ⚠️ 矛盾：...（来源 vs 来源）` 块，让人类裁决

**经验法则**：一份新资料通常牵动 5–15 个 wiki 页面更新。

**但有两个合法例外** ⭐ 新增 2026-05-26（反信息茧房原则）：
- **种子节点**：跨域弱关联但内容扎实的资料，可以只建 1–3 个文件，标 `#种子` tag，等未来其他 ingest 出现交叉点时再生长。**不要因为"和现有内容不共振"就放弃录入有启发的跨域内容**（LLM Wiki 本身就是 Karpathy 跨 Memex/Zettelkasten/REM 睡眠拼出来的）
- **资料确实太薄**：评估后明确告知人类"资料含金量不足以独立成页"，不要凑数

**真正反对的不是"不共振"，是水文**：营销稿、空泛预测、二次转载、标题党、重复已知

**ingest 完毕回头检查"有没有 surprise"** ⭐ 新增 2026-05-26：
如果整个过程平淡无奇、没遇到任何意外发现，**很可能是只看了二手稿、被框架化叙述带跑了**。强迫自己问一句"我漏掉了什么？"

### 2.2 Query — 提问

触发：人类对知识库提问

步骤：
1. **先读 `wiki/index.md`**：定位相关页面
2. **读相关 wiki 页**：综合回答（不是检索原文片段）
3. **回答里必须带 `[[页面链接]]`**：让人类能跳过去看
4. **沉淀回 wiki**：如果回答中产生了新的综合/对比/洞察，**必须**写进 `wiki/syntheses/` 或更新相关概念页。**问完即忘 = 失败**
5. **追加 log.md**：`YYYY-MM-DD HH:MM | query | <问题摘要> | 新增/更新 N 个页面`

### 2.3 Maintain — 审计维护

触发：
- 人类说"审计一下 / 体检一下 / maintain"
- 每录入 ~10 份资料后主动建议
- **Dreams 式主动建议** ⭐ 新增 2026-05-26：累计 ≥5 次 ingest 或距离上次 maintain >14 天 → 主动**建议**人类做审计（不自动执行，决策权留人）。灵感来自 [[concepts/REM 睡眠式记忆整合]]

输出一份审计报告（写到 `wiki/syntheses/YYYYMMDD-审计报告.md`），逐项检查：

- [ ] **矛盾**：页面之间是否有未裁决的冲突
- [ ] **过时**：被新资料取代的旧结论
- [ ] **孤儿页**：没有任何入链的页面
- [ ] **缺失反链**：A 提到 B 但 B 页没反链回 A
- [ ] **隐藏概念**：在多处被提到但没有独立页的重要术语
- [ ] **空壳页**：只有标题、内容寡淡的页
- [ ] **研究建议**：根据现有 Wiki 的形状，建议人类去找什么新资料、问什么新问题

修复能改的，改不了的列出来等人类决策。

## 3. 命名与格式规范

### 3.1 文件命名

| 类型 | 格式 | 例 |
|---|---|---|
| raw 文件 | `YYYYMMDD-来源-标题.{md,pdf,...}` | `20260524-知乎-Karpathy开源Agent.md` |
| 摘要页 | 同上，放 `wiki/summaries/` | `20260524-知乎-Karpathy开源Agent.md` |
| 实体页 | `实体名.md`（中文用中文，英文用英文原名） | `Andrej Karpathy.md`、`Anthropic.md` |
| 概念页 | `概念名.md` | `LLM Wiki.md`、`渐进式披露.md` |
| 综述页 | `YYYYMMDD-主题.md` | `20260524-RAG vs LLM Wiki对比.md` |
| 审计报告 | `YYYYMMDD-审计报告.md` | `20260524-审计报告.md` |

### 3.2 Frontmatter（每个 wiki 页头部必带）

```yaml
---
type: entity | concept | summary | synthesis
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources:
  - "[[20260524-知乎-XXX]]"
source_tier: 一手源 | 二手源    # 仅 summary 必填，⭐ 新增 2026-05-26
tags: [ai, 知识管理]
aliases: [别名1, 别名2]
---
```

**数字事实必标时效** ⭐ 新增 2026-05-26：
任何会变化的数字（star 数、用户数、估值、版本号、市场份额等）必须带 `(截至 YYYY-MM-DD)` 后缀。

例：`GitHub **15.5 万 star**(截至 2026-05-26)`

避免知识库出现"永远过时但不知道何时过时"的鬼数据。

### 3.3 链接

- **双向链**：一律用 `[[页面名]]`，不要用 markdown 链接
- **来源标注**：陈述事实后用 `(来源: [[摘要页名]])` 标注
- **批量更新时**：宁可多链不要少链

### 3.4 Tag 体系（开放，可生长）

种子标签（按需扩展）：
- 领域：`#ai` `#llm` `#agent` `#电商` `#运动户外` `#知识管理`
- 类型：`#方法论` `#工具` `#案例` `#人物` `#公司`
- 状态：`#待深入` `#存疑` `#已验证` `#待验证`
- 信源：`#一手源` `#二手源` ⭐ 新增 2026-05-26
- 生长状态：`#种子`（孤立但有启发，待未来生长）⭐ 新增 2026-05-26

新标签随时加，但加完后要去 `wiki/index.md` 的 "Tag 一览" 区登记一下。

## 4. 关键约束

1. **绝不删除 raw/**：哪怕原文有错，也只在 wiki 层批注
2. **wiki 页改写要追加来源**：不要无来源地添加新论断
3. **每次操作都要写 log.md**：审计的根基
4. **不确定时停下来问人类**：不要为了"看起来勤快"瞎写
5. **不要污染先生的工作 vault**：所有写操作严格限制在 `【K】Karpathy知识库/` 目录内
6. **二手必溯一手** ⭐ 新增 2026-05-26：遇到二手源必须主动追溯到一手，溯不到则明确标 `#二手源`
7. **反信息茧房** ⭐ 新增 2026-05-26：欢迎跨域弱关联的种子节点（标 `#种子`），反对仅因"不共振"而拒绝有启发的内容。真正要拒的是水文（营销稿/空泛预测/二次转载/标题党）
8. **警惕框架化叙述** ⭐ 新增 2026-05-26：二手源若用某概念"套"新事物（如"这就是 X 的 Y"），会过滤掉框架外的关键信息——遇到这类叙述要更警觉地溯源

## 5. 迭代本协议

如果你（Agent）在使用中发现本协议有 bug、漏洞、或可改进之处：

1. 不要自作主张修改
2. 在 `log.md` 顶部写一行 `YYYY-MM-DD HH:MM | proposal | <改进建议>`
3. 等人类批准后再改 `AGENT.md`
4. 改完在 log.md 写 `YYYY-MM-DD HH:MM | agent-update | <改了什么>`
